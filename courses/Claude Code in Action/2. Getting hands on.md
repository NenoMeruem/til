# Claude Code setup


Đã đến lúc thiết lập **Claude Code** trên máy local của bạn! Bạn có thể xem hướng dẫn chi tiết tại đây:  
[https://code.claude.com/docs/en/quickstart](https://code.claude.com/docs/en/quickstart)

Dưới đây là các bước cài đặt tóm tắt, sử dụng đúng thuật ngữ chuyên ngành để bạn dễ thao tác:

## Cài đặt Claude Code

Tùy theo hệ điều hành (OS) và môi trường runtime bạn đang sử dụng, hãy chọn lệnh phù hợp:

### macOS (sử dụng Homebrew)

Nếu bạn đã cài **Homebrew** (trình quản lý package cho macOS), chạy lệnh sau trong terminal:

```bash
brew install --cask claude-code
```

- `--cask` dùng để cài đặt ứng dụng dạng binary.
- Lệnh này sẽ pull package từ repository và cài đặt vào hệ thống.

### macOS, Linux, hoặc WSL

Chạy script cài đặt tự động bằng `curl`:

```bash
curl -fsSL https://claude.ai/install.sh | bash
```

Giải thích nhanh:

- `curl` dùng để tải shell script từ remote server.
- `-fsSL` giúp fail nhanh nếu có lỗi và ẩn progress output.
- Dữ liệu tải về được pipe (`|`) trực tiếp vào `bash` để thực thi.

### Windows (CMD)

Chạy lệnh sau trong Command Prompt:

```bash
curl -fsSL https://claude.ai/install.cmd -o install.cmd && install.cmd && del install.cmd
```

Quy trình:

1. Download file `install.cmd`
2. Execute script cài đặt
3. Xóa file sau khi hoàn tất (cleanup)

## Khởi chạy CLI

Sau khi cài đặt xong, mở terminal và chạy:

```bash
claude
```

- Ở lần chạy đầu tiên, CLI sẽ yêu cầu bạn **authenticate** (xác thực tài khoản).
- Quá trình này thường mở một browser session để bạn login và cấp quyền truy cập.

## Cấu hình bổ sung (nếu dùng Cloud Provider)

Nếu bạn tích hợp Claude Code với các nền tảng cloud như **AWS Bedrock** hoặc **Google Cloud Vertex AI**, bạn cần thực hiện thêm bước cấu hình.

### AWS Bedrock

Xem hướng dẫn chi tiết tại:  
[https://code.claude.com/docs/en/amazon-bedrock](https://code.claude.com/docs/en/amazon-bedrock)

Bạn sẽ cần:

- Cấu hình IAM permissions
- Thiết lập AWS credentials (qua AWS CLI hoặc environment variables)
- Chọn đúng region hỗ trợ Bedrock

### Google Cloud Vertex AI

Hướng dẫn tại:  
[https://code.claude.com/docs/en/google-vertex-ai](https://code.claude.com/docs/en/google-vertex-ai)

Bạn sẽ cần:
- Enable Vertex AI API
- Thiết lập service account
- Cấu hình authentication qua `gcloud` CLI
- Export biến môi trường như `GOOGLE_APPLICATION_CREDENTIALS`

# Project setup

Làm việc với **Claude Code** sẽ thú vị hơn nhiều khi bạn có một **project thực tế** để thao tác. Tôi đã chuẩn bị sẵn một project nhỏ để bạn có thể dùng thử cùng Claude Code. Đây chính là ứng dụng **tạo UI tự động (UI generation app)** đã được demo trong video trước.

> Lưu ý: Bạn **không bắt buộc** phải chạy project này. Bạn hoàn toàn có thể tiếp tục khóa học bằng cách sử dụng **codebase riêng** của mình nếu muốn.


## Yêu cầu cài đặt (Setup)

Project này cần một số bước cấu hình ban đầu như sau:

### 1. Cài đặt môi trường runtime

Đảm bảo bạn đã cài đặt **Node.js** trên máy local.

Node.js là môi trường runtime cho JavaScript phía server, đồng thời cung cấp **npm (Node Package Manager)** để quản lý dependencies.

### 2. Tải source code

- Tải file `uigen.zip` được đính kèm trong bài giảng.
- Giải nén (extract) file này vào một thư mục bất kỳ trên máy của bạn.
- Mở terminal và di chuyển vào thư mục project.

### 3. Cài đặt dependencies & khởi tạo database

Trong thư mục project, chạy lệnh:

```bash
npm run setup
```

Lệnh này sẽ:

- Cài đặt toàn bộ **dependencies** được khai báo trong `package.json`
- Thiết lập một **local SQLite database**

Project sử dụng **SQLite**, một hệ quản trị cơ sở dữ liệu quan hệ dạng file-based, nhẹ và không cần server riêng.

## (Tùy chọn) Cấu hình API để sử dụng Claude thật

Project này sử dụng Claude thông qua **Anthropic API** để sinh tự động các UI component. Nếu bạn muốn test đầy đủ chức năng sinh code bằng AI, bạn cần cung cấp **API key**.

> Việc này là tùy chọn.  
> Nếu không có API key, ứng dụng vẫn hoạt động nhưng chỉ sinh ra **mock code tĩnh (static fake code)**.

### Cách cấu hình API key

1. Truy cập:  [https://console.anthropic.com/](https://console.anthropic.com/)
2. Tạo một API key trong dashboard của **Anthropic**
3. Mở file `.env` trong project và thêm:

```env
ANTHROPIC_API_KEY=your_api_key_here
```

`.env` là file dùng để cấu hình **environment variables** cho ứng dụng.

## 4. Chạy ứng dụng ở môi trường development

Sau khi hoàn tất setup, chạy:

```bash
npm run dev
```

Lệnh này sẽ:
- Khởi động **development server**
- Enable hot-reload (nếu project có cấu hình)
- Cho phép bạn test app trên môi trường local

# Adding context

Khi làm việc với Claude trong các dự án lập trình, **quản lý ngữ cảnh (context management)** là yếu tố then chốt.

Một dự án phần mềm có thể bao gồm hàng chục, thậm chí hàng trăm file (source code, config, tài liệu, test case...). Tuy nhiên, Claude không cần toàn bộ codebase để hỗ trợ bạn hiệu quả — nó chỉ cần **đúng thông tin liên quan (relevant context)** đến vấn đề đang xử lý.

Việc cung cấp quá nhiều ngữ cảnh không liên quan (irrelevant context) thực tế còn làm giảm hiệu suất xử lý của Claude. Điều này có thể dẫn đến:

- Phản hồi kém chính xác hơn do “nhiễu thông tin”
- Phân tích sai phạm vi (scope) của vấn đề
- Đề xuất giải pháp không bám sát module hoặc component cần chỉnh sửa

Vì vậy, bạn cần học cách:
- Xác định đúng **file, module hoặc function** liên quan trực tiếp đến issue
- Cung cấp các đoạn code (code snippet) thay vì toàn bộ file nếu không cần thiết
- Chỉ đính kèm tài liệu hoặc API spec thực sự phục vụ cho tác vụ hiện tại
- Làm rõ bối cảnh như stack công nghệ, version framework, hoặc kiến trúc hệ thống khi cần

Tóm lại, thay vì “dump” toàn bộ project vào prompt, hãy tối ưu hóa input theo hướng **tập trung, có chọn lọc và định hướng rõ ràng**. Điều này giúp Claude hiểu đúng vấn đề, giảm độ nhiễu và đưa ra giải pháp chính xác, tối ưu hơn cho workflow phát triển phần mềm của bạn.

### The /init Command
Khi bạn khởi động **Claude** lần đầu trong một **project mới**, hãy chạy lệnh `/init`.
Lệnh này yêu cầu Claude thực hiện quá trình **phân tích toàn bộ codebase** để nắm bắt các yếu tố cốt lõi sau:
- **Mục đích (purpose)** và **kiến trúc hệ thống (architecture)** của dự án
- Các **command quan trọng**, script chính và những **file critical**
- **Coding patterns**, quy ước đặt tên (naming conventions) và cấu trúc thư mục (project structure)

![Image](images/Getting%20hands%20on/Pasted%20image%2020260302111943.png)

Sau khi hoàn tất quá trình scan và phân tích static code, Claude sẽ:
1. Tạo một bản **tổng hợp (summary)** về codebase
2. Ghi nội dung đó vào file `CLAUDE.md` ở thư mục gốc của dự án
File `CLAUDE.md` đóng vai trò như một **metadata document**, giúp Claude và các developer khác hiểu nhanh bối cảnh kỹ thuật của project trong các phiên làm việc tiếp theo.

Lệnh `/init` giúp:
- Bootstrap môi trường làm việc
- Index và phân tích toàn bộ codebase
- Thiết lập ngữ cảnh (context) cho Claude
- Tạo tài liệu hệ thống tự động (`CLAUDE.md`)
Đây là bước khởi tạo quan trọng để Claude có đủ context trước khi bạn bắt đầu refactor, debug hoặc phát triển tính năng mới.


### File **CLAUDE.md**

Tệp **CLAUDE.md** đóng vai trò như một **tài liệu cấu hình (configuration document)** dành cho Claude trong phạm vi codebase của bạn. Nó phục vụ hai mục đích chính:
1. **Định hướng Claude trong codebase**    
    - Chỉ ra các **command quan trọng** (build, test, deploy, lint, v.v.)
    - Mô tả **kiến trúc hệ thống (system architecture)**
    - Quy định **coding conventions / coding style** mà dự án đang áp dụng
    Điều này giúp Claude hiểu rõ cấu trúc dự án, luồng xử lý (workflow) và tiêu chuẩn kỹ thuật trước khi thực hiện bất kỳ tác vụ nào như refactor, debug hay generate code.
2. **Cung cấp chỉ thị tùy chỉnh (custom instructions)**  
    Bạn có thể thêm các **directive riêng** cho Claude, ví dụ:
    - Quy tắc đặt tên biến (naming conventions)
    - Chuẩn commit message
    - Yêu cầu về bảo mật (security constraints)
    - Giới hạn phạm vi chỉnh sửa file

Tệp **CLAUDE.md** sẽ được **inject vào mọi request** bạn gửi tới Claude.  Về bản chất, nó hoạt động tương tự như một **persistent system prompt** ở cấp độ project.
Điều này có nghĩa là:
- Claude luôn “nhớ” các quy ước và cấu trúc dự án
- Không cần lặp lại hướng dẫn trong mỗi lần gọi
- Giảm sai lệch ngữ cảnh (context drift) khi làm việc với codebase lớn
Nó giúp tối ưu hóa quá trình tương tác giữa developer và AI, đảm bảo tính nhất quán (consistency), tuân thủ kiến trúc (architecture compliance) và chuẩn hóa coding style trong toàn bộ vòng đời phát triển phần mềm (SDLC).

### CLAUDE.md File Locations

Claude nhận diện **ba file cấu hình `CLAUDE.md`** ở ba vị trí phổ biến trong hệ thống. Mỗi file có phạm vi (scope) và mục đích sử dụng khác nhau:

![Image](images/Getting%20hands%20on/Pasted%20image%2020260302112417.png)

#### 1. `CLAUDE.md`
- Được khởi tạo bằng lệnh `/init`.
- Được **commit vào source control** (ví dụ: Git repository).
- Được **chia sẻ cho toàn bộ team/engineer** trong dự án.
- Thường dùng để định nghĩa **quy ước chung, coding standards, workflow, guideline** mà Claude cần tuân theo khi làm việc trong project đó.

Đây là file cấu hình ở **cấp độ project (project-level configuration)**.
#### 2. `CLAUDE.local.md`
- **Không được chia sẻ** với các engineer khác.
- Không được commit lên repository (thường nằm trong `.gitignore`).
- Chứa các **instruction cá nhân, custom rule, preference riêng** của từng developer khi sử dụng Claude.

Đây là file cấu hình ở **cấp độ local workspace (local-only configuration)**.
#### 3. `~/.claude/CLAUDE.md`
- Nằm trong thư mục home của user (`~/.claude/`).
- Được áp dụng cho **tất cả các project trên máy**.
- Dùng để cấu hình các **global rule, mặc định hệ thống, behavior chung** mà bạn muốn Claude luôn tuân theo.

Đây là file cấu hình ở **cấp độ global (machine-wide configuration)**.

### Adding Custom Instructions

Bạn có thể tùy chỉnh cách Claude hoạt động bằng cách thêm các instruction vào file `CLAUDE.md`.

Ví dụ: nếu Claude đang thêm quá nhiều comment vào source code, bạn có thể điều chỉnh hành vi này bằng cách update lại file cấu hình.

Sử dụng command `#` để vào **memory mode** — chế độ này cho phép bạn chỉnh sửa file `CLAUDE.md` một cách thông minh.

Chỉ cần nhập một câu lệnh như:

```
# Use comments sparingly. Only comment complex code.
```

Claude sẽ tự động merge (hợp nhất) instruction này vào file `CLAUDE.md` của bạn.

### File Mentions with '@'

Khi bạn cần Claude phân tích hoặc tham chiếu đến các tệp (file) cụ thể trong dự án, hãy sử dụng ký hiệu `@` theo sau là **đường dẫn tệp (file path)**. Cơ chế này sẽ tự động inject nội dung của tệp đó vào ngữ cảnh (context) của request, giúp Claude có đầy đủ dữ liệu để phân tích.

Ví dụ, nếu bạn muốn tìm hiểu về hệ thống xác thực (authentication system) và biết các tệp liên quan nằm trong thư mục `auth`, bạn có thể nhập:

```
How does the auth system work? @auth
```

![Image](images/Getting%20hands%20on/Pasted%20image%2020260302143414.png)


Khi đó, Claude sẽ:
1. Truy vấn và hiển thị danh sách các tệp liên quan trong thư mục `auth`.
2. Cho phép bạn chọn tệp phù hợp.
3. Tự động load nội dung của tệp đã chọn vào phiên làm việc (session).
4. Sử dụng nội dung này làm ngữ cảnh để phân tích, giải thích luồng xử lý (flow), kiến trúc (architecture) hoặc logic nghiệp vụ (business logic).

#### Cách hoạt động (về mặt kỹ thuật)
- `@<file_path>` đóng vai trò như một **context reference directive**.
- Hệ thống sẽ thực hiện một thao tác tương tự như context binding, gắn nội dung file vào prompt trước khi gửi đến model.
- Điều này giúp giảm ambiguity và tăng độ chính xác khi Claude cần:
    - Phân tích source code
    - Debug
    - Review code
    - Giải thích kiến trúc hệ thống
    - Truy vết dependency giữa các module
#### Lợi ích
- Giảm việc copy/paste thủ công.
- Đảm bảo Claude làm việc trên đúng phiên bản source code.
- Tăng tính chính xác khi phân tích các thành phần như controller, service, middleware, hoặc config.

Tóm lại, sử dụng `@` là cách tối ưu để cung cấp ngữ cảnh chính xác cho Claude khi làm việc với codebase hoặc các tài nguyên cụ thể trong dự án.

### Referencing Files in CLAUDE.md

Bạn cũng có thể **tham chiếu trực tiếp các file** bên trong file `CLAUDE.md` bằng cú pháp `@` tương tự như khi import hoặc link dependency. Cách này đặc biệt hữu ích đối với những file đóng vai trò **core resource** và được sử dụng xuyên suốt nhiều module trong project.

Ví dụ, nếu bạn có một file định nghĩa **database schema** mô tả cấu trúc dữ liệu của hệ thống, bạn có thể thêm vào `CLAUDE.md` nội dung như sau:

> Database schema được định nghĩa trong file `@prisma/schema.prisma`.  
> Hãy tham chiếu file này bất cứ khi nào cần phân tích hoặc hiểu cấu trúc dữ liệu được lưu trữ trong database.
#### Cơ chế hoạt động
Khi bạn tham chiếu file bằng cú pháp `@filename`, hệ thống sẽ:
- **Tự động inject nội dung của file đó vào context của mỗi request**
- Loại bỏ nhu cầu phải đọc lại file thủ công
- Giúp Claude truy cập trực tiếp vào cấu trúc dữ liệu, model, hoặc config liên quan

![Image](images/Getting%20hands%20on/Pasted%20image%2020260302150852.png)

Điều này tương đương với việc preload một dependency vào runtime context.

#### Lợi ích trong workflow phát triển
- Giảm overhead khi phải re-parse schema nhiều lần
- Tăng tốc độ phản hồi vì không cần search & load file lại
- Đảm bảo tính nhất quán khi làm việc với data model
- Hữu ích cho các file có tính nền tảng như:
    - Database schema
    - API contract (OpenAPI/Swagger)
    - Type definitions
    - System architecture config

Việc sử dụng cú pháp `@` trong `CLAUDE.md` giống như khai báo một **global reference** cho project. Nội dung file được tự động đưa vào execution context của mọi request, giúp Claude hiểu rõ cấu trúc hệ thống ngay lập tức mà không cần thao tác bổ sung.

Đây là một best practice khi làm việc với các project có kiến trúc phức tạp hoặc data model lớn.

## Making changes

Khi làm việc với **Claude** trong môi trường phát triển (development environment), bạn sẽ thường xuyên phải **chỉnh sửa và mở rộng các project hiện có** thay vì xây dựng từ đầu. Tài liệu này trình bày các kỹ thuật thực tiễn (best practices) giúp bạn triển khai thay đổi một cách hiệu quả, tối ưu quy trình làm việc và giảm thiểu rủi ro phát sinh lỗi.

Cụ thể, nội dung bao gồm:
- **Cách implement thay đổi một cách có hệ thống**: phân tích yêu cầu, refactor code, đảm bảo backward compatibility và duy trì tính nhất quán của codebase.
- **Sử dụng giao tiếp trực quan (visual communication)** thông qua screenshot để cung cấp thêm context về UI, layout, bug hiển thị hoặc trạng thái runtime — giúp Claude hiểu chính xác vấn đề trong quá trình debug hoặc cải tiến tính năng.
- **Tận dụng khả năng suy luận nâng cao (advanced reasoning capabilities)** của Claude để phân tích dependency, đề xuất kiến trúc phù hợp, tối ưu thuật toán, hoặc xử lý các logic phức tạp trong hệ thống.

Mục tiêu là giúp bạn tích hợp Claude vào workflow phát triển phần mềm một cách hiệu quả, nâng cao năng suất lập trình và đảm bảo chất lượng sản phẩm.

#### Using Screenshots for Precise Communication

Một trong những cách **hiệu quả nhất để tương tác với Claude** là sử dụng **screenshot (ảnh chụp màn hình)**.

Khi bạn muốn **chỉnh sửa một thành phần cụ thể trong UI (User Interface)** của ứng dụng, việc cung cấp screenshot sẽ giúp Claude xác định chính xác **component, layout, hoặc vùng DOM** mà bạn đang đề cập đến. Điều này giúp giảm ambiguity và tăng độ chính xác khi mô tả yêu cầu.

Để **paste screenshot vào giao diện chat của Claude**, hãy sử dụng tổ hợp phím: **Ctrl + V**  _(Lưu ý: trên macOS vẫn dùng Ctrl + V, không dùng Cmd + V trong trường hợp này.)_

Shortcut này được thiết kế riêng để inject trực tiếp image data vào chat interface.
Sau khi image được upload thành công, bạn có thể:
- Chỉ định rõ component cần refactor
- Yêu cầu chỉnh sửa UI/UX
- Thay đổi style (CSS), layout, spacing
- Cập nhật logic hiển thị hoặc behavior

Việc cung cấp screenshot giúp Claude có thêm **visual context**, từ đó phân tích chính xác cấu trúc giao diện và đề xuất giải pháp phù hợp hơn cho application của bạn.


#### Planning Mode

Đối với các tác vụ phức tạp cần **nghiên cứu sâu (deep dive)** trên toàn bộ **codebase**, bạn có thể kích hoạt **Planning Mode**. Tính năng này cho phép Claude thực hiện quá trình **phân tích và khám phá hệ thống một cách toàn diện** trước khi bắt đầu triển khai thay đổi.

Bạn có thể bật Planning Mode bằng cách nhấn **Shift + Tab hai lần** (hoặc một lần nếu bạn đang bật chế độ **auto-accept edits**).

Khi ở trong Planning Mode, Claude sẽ:

- Đọc và phân tích nhiều file hơn trong project (bao gồm các module, dependency liên quan)
- Xây dựng một **implementation plan** chi tiết (kế hoạch triển khai cụ thể)
- Trình bày rõ ràng những thay đổi dự kiến sẽ thực hiện (các bước refactor, cập nhật logic, chỉnh sửa cấu trúc, v.v.)
- Tạm dừng và chờ bạn **phê duyệt (approval)** trước khi tiến hành chỉnh sửa mã nguồn

Cơ chế này giúp bạn:

- Review trước kiến trúc giải pháp đề xuất
- Phát hiện các edge case hoặc scenario đặc biệt có thể bị bỏ sót
- Điều chỉnh hướng tiếp cận nếu cần (ví dụ: thay đổi design pattern, tối ưu performance, hoặc đảm bảo backward compatibility)

Tóm lại, Planning Mode đóng vai trò như một bước **pre-implementation review**, giúp giảm rủi ro, hạn chế regression và đảm bảo thay đổi được thực hiện một cách có kiểm soát trong toàn bộ hệ thống.

#### Thinking Modes

Claude cung cấp nhiều cấp độ suy luận (_reasoning levels_) thông qua các chế độ “thinking”. Những chế độ này cho phép hệ thống phân bổ nhiều tài nguyên tính toán hơn (compute budget) và sử dụng nhiều _token_ hơn để phân tích các bài toán phức tạp trước khi sinh ra câu trả lời (_response generation_).

Các chế độ tư duy hiện có bao gồm:

- **“Think”** – Suy luận cơ bản (_basic reasoning_), phù hợp với các bài toán đơn giản hoặc yêu cầu xử lý logic ngắn gọn.
- **“Think more”** – Suy luận mở rộng (_extended reasoning_), cho phép phân tích sâu hơn với không gian ngữ cảnh (_context window_) lớn hơn.
- **“Think a lot”** – Suy luận toàn diện (_comprehensive reasoning_), tối ưu cho các vấn đề đa bước (_multi-step problems_).
- **“Think longer”** – Suy luận với thời gian xử lý dài hơn (_extended time reasoning_), tăng khả năng đánh giá nhiều hướng tiếp cận khác nhau trước khi đưa ra kết luận.
- **“Ultrathink”** – Mức suy luận tối đa (_maximum reasoning capability_), sử dụng nhiều token nhất để thực hiện phân tích chuyên sâu (_deep analysis_) và giải quyết các bài toán có độ phức tạp cao.

Mỗi chế độ sẽ cấp phát số lượng _token_ ngày càng nhiều cho mô hình, từ đó cho phép thực hiện phân tích chuyên sâu hơn (_deeper analysis_), xử lý tốt hơn các bài toán có nhiều ràng buộc (_constraints_) hoặc yêu cầu suy luận đa tầng (_hierarchical reasoning_).


![Image](images/Getting%20hands%20on/Pasted%20image%2020260302152248.png)


#### When to Use Planning vs Thinking

Hai tính năng này được thiết kế để xử lý các mức độ phức tạp khác nhau trong quá trình phát triển phần mềm:

##### **Planning Mode** – Phù hợp với các tác vụ cần góc nhìn tổng thể (high-level view)

Nên sử dụng khi:
- Yêu cầu **hiểu toàn bộ codebase** hoặc kiến trúc hệ thống
- Triển khai **tính năng nhiều bước (multi-step implementation)**
- Thực hiện các thay đổi ảnh hưởng đến **nhiều file, module hoặc component**
- Refactor liên quan đến dependency giữa nhiều phần của hệ thống
- Thiết kế lại kiến trúc (architecture redesign) hoặc tối ưu cấu trúc dự án

Tóm lại, Planning Mode phù hợp với các bài toán mang tính **chiến lược (strategic level)** và cần cái nhìn **toàn cảnh (macro-level analysis)**.

##### **Thinking Mode** – Phù hợp với các tác vụ cần đào sâu logic (deep reasoning)

Nên sử dụng khi:
- Giải quyết các vấn đề **logic phức tạp**
- **Debug** những lỗi khó tái hiện hoặc khó truy vết
- Xử lý các bài toán liên quan đến **thuật toán (algorithmic challenges)**
- Tối ưu performance ở mức implementation
- Phân tích luồng xử lý (control flow) hoặc cấu trúc dữ liệu phức tạp

Thinking Mode phù hợp với các bài toán mang tính **chi tiết (low-level / implementation-level)** và cần suy luận chuyên sâu.

Có thể kết hợp cả hai chế độ. Trong thực tế, nhiều tác vụ yêu cầu cả:
- **Chiều rộng (breadth)** → hiểu hệ thống tổng thể
- **Chiều sâu (depth)** → xử lý logic cụ thể

Bạn hoàn toàn có thể kết hợp cả Planning Mode và Thinking Mode để giải quyết các bài toán vừa mang tính kiến trúc vừa mang tính thuật toán.

##### Lưu ý về chi phí

Cả hai chế độ đều tiêu tốn **nhiều token hơn** do cần phân tích và suy luận sâu hơn.  
Điều này đồng nghĩa với việc:
- Tăng chi phí sử dụng
- Tăng thời gian xử lý
Vì vậy, nên cân nhắc sử dụng khi thực sự cần thiết, đặc biệt trong các hệ thống production hoặc môi trường có giới hạn tài nguyên.

# Controlling context

Khi làm việc với Claude trong các tác vụ phức tạp, bạn thường cần **điều hướng hội thoại (conversation steering)** để giữ cho quá trình trao đổi luôn **tập trung (focus)** và **hiệu quả (productivity)**.

Có nhiều kỹ thuật bạn có thể áp dụng để **kiểm soát luồng hội thoại (conversation flow control)** và giúp Claude luôn bám sát mục tiêu ban đầu, tránh bị lệch ngữ cảnh (context drift) hoặc sinh nội dung ngoài phạm vi yêu cầu.

Nói cách khác, bạn nên chủ động đóng vai trò như một **orchestrator** trong quá trình tương tác, sử dụng các chiến lược như:
- Chia nhỏ bài toán (task decomposition)
- Ràng buộc phạm vi (scope constraint)
- Làm rõ yêu cầu đầu vào/đầu ra (input/output specification)
- Nhắc lại mục tiêu khi cần (goal reinforcement)

Việc này giúp tối ưu hóa chất lượng phản hồi, giảm thiểu nhiễu (noise), và đảm bảo mô hình luôn hoạt động đúng theo **intent** của bạn.

## Interrupting Claude with Escape

Đôi khi **Claude** bắt đầu đi lệch hướng hoặc cố gắng xử lý quá nhiều tác vụ cùng lúc. Bạn có thể nhấn phím **Escape (ESC)** để dừng phản hồi của Claude giữa chừng, từ đó điều hướng lại cuộc hội thoại theo đúng mục tiêu mong muốn.

Cơ chế này đặc biệt hữu ích khi bạn muốn Claude tập trung vào **một tác vụ cụ thể**, thay vì xử lý đồng thời nhiều yêu cầu dẫn đến phản hồi quá dài hoặc vượt ngoài phạm vi mong đợi.

Ví dụ, nếu bạn yêu cầu Claude viết **unit test** cho nhiều hàm (functions) và nó bắt đầu xây dựng một **test strategy tổng thể** cho toàn bộ hệ thống, bạn có thể ngắt phản hồi và yêu cầu nó chỉ tập trung vào **một function duy nhất** tại một thời điểm.

## Combining Escape with Memories

Một trong những ứng dụng mạnh mẽ nhất của **kỹ thuật escape (thoát tiến trình phản hồi)** là xử lý các **lỗi lặp lại (repetitive errors)**.

Khi Claude liên tục mắc cùng một lỗi ở nhiều cuộc hội thoại khác nhau, bạn có thể thực hiện quy trình sau:

1. Nhấn **Escape** để **interrupt (ngắt tiến trình sinh phản hồi hiện tại)**.
2. Sử dụng phím tắt **#** để thêm một **memory (bản ghi ngữ cảnh dài hạn / persistent context)** về cách xử lý đúng.
3. Tiếp tục cuộc hội thoại với **thông tin đã được hiệu chỉnh (corrected information)**.

![[Pasted image 20260303161337.png]]

Cách làm này giúp hệ thống **cập nhật ngữ cảnh dự án (project-level context)** và tránh lặp lại cùng một lỗi trong các cuộc hội thoại tương lai. Nói cách khác, bạn đang chủ động **ghi đè hành vi sai bằng một cấu hình hướng dẫn chính xác**, giúp cải thiện tính nhất quán và độ chính xác của mô hình theo thời gian.

## Rewinding Conversations

Trong các cuộc hội thoại dài, bạn có thể **tích lũy quá nhiều context**, bao gồm cả những thông tin không còn liên quan hoặc gây nhiễu cho tác vụ hiện tại.

Ví dụ: nếu Claude gặp một **runtime error** và bạn phải thực hiện quá trình **debugging** qua nhiều lượt trao đổi (back-and-forth), thì toàn bộ phần thảo luận đó có thể trở nên dư thừa đối với nhiệm vụ tiếp theo. Việc giữ lại những đoạn context này có thể làm giảm hiệu quả xử lý, vì mô hình vẫn phải parse và cân nhắc chúng trong quá trình suy luận.

![[Pasted image 20260303161817.png]]

Để giải quyết vấn đề này, bạn có thể **rewind conversation state** bằng cách nhấn phím **Escape hai lần**. Thao tác này sẽ hiển thị toàn bộ message history mà bạn đã gửi, cho phép bạn:
- Quay về một checkpoint trước đó trong luồng hội thoại
- Tiếp tục tương tác từ một trạng thái context sạch hơn
- Loại bỏ những đoạn trao đổi không còn cần thiết

![[Pasted image 20260303161854.png]]

Kỹ thuật này giúp bạn:
- Giữ lại những context quan trọng (ví dụ: hiểu biết của Claude về codebase, kiến trúc hệ thống, hoặc yêu cầu nghiệp vụ)
- Loại bỏ conversation history gây nhiễu hoặc không còn liên quan
- Giúp mô hình tập trung vào current task, giảm “context pollution”

Tóm lại, đây là một cách quản lý **context window** hiệu quả, đặc biệt hữu ích khi làm việc với các tác vụ kỹ thuật như lập trình, phân tích hệ thống hoặc xử lý lỗi phức tạp.

## Context Management Commands

Anthropic cung cấp trong Claude một số **command** giúp bạn quản lý **conversation context** hiệu quả hơn trong quá trình làm việc:

#### `/compact`

Lệnh `/compact` thực hiện **context summarization** cho toàn bộ **conversation history**, đồng thời vẫn giữ lại những **key insights**, thông tin quan trọng và các ràng buộc mà Claude đã học được từ phiên làm việc trước đó.
### Use case phù hợp:
- Claude đã thu thập được nhiều thông tin quan trọng về **project architecture**, **business logic**, hoặc **system requirements** của bạn.
- Bạn muốn tiếp tục triển khai các **related tasks** (ví dụ: refactor code, mở rộng feature, tối ưu performance).
- Cuộc hội thoại đã dài, context window gần đầy nhưng vẫn chứa nhiều dữ liệu quan trọng cần preserve.

Nói cách khác, đây là thao tác **context compression** nhưng không làm mất các thông tin trọng yếu phục vụ cho workflow hiện tại.

#### `/clear`

Lệnh `/clear` thực hiện **hard reset** toàn bộ conversation context, xóa sạch lịch sử tương tác trước đó.

### Use case phù hợp:
- Bạn chuyển sang một **task hoàn toàn khác**, không liên quan đến nội dung trước.
- Context cũ có thể gây **context leakage** hoặc làm model suy luận sai hướng.
- Bạn muốn khởi tạo một phiên làm việc mới với **clean state**.

## Khi nào nên sử dụng các kỹ thuật này

Các kỹ thuật **điều khiển hội thoại (conversation control)** này đặc biệt hữu ích trong các trường hợp sau:
- **Các cuộc hội thoại kéo dài (long-running conversations)**, nơi _context_ ngày càng “phình to” và gây nhiễu bộ nhớ ngữ cảnh (context window).
- **Chuyển đổi tác vụ (task transitions)**, khi ngữ cảnh cũ có thể làm phân tán trọng tâm của tác vụ mới.
- **Tình huống Claude lặp lại cùng một lỗi (repeated failure patterns)** do bị “neo” vào ngữ cảnh trước đó.
- **Các dự án phức tạp (complex projects)** yêu cầu duy trì sự tập trung vào một module, component hoặc scope cụ thể.

Bằng cách sử dụng chiến lược các lệnh như **escape**, **double-tap escape**, **/compact**, và **/clear**, bạn có thể:

- Kiểm soát và tối ưu hóa _context management_
- Giảm _noise_ trong phiên làm việc
- Tránh hiện tượng _context leakage_
- Cải thiện độ chính xác của phản hồi
- Duy trì hiệu suất làm việc ổn định trong _development workflow_

Đây không chỉ là các tính năng tiện lợi (_convenience features_), mà thực chất là những công cụ thiết yếu giúp duy trì các phiên **AI-assisted development** hiệu quả, đặc biệt khi làm việc với hệ thống có giới hạn _token window_ và phụ thuộc mạnh vào ngữ cảnh hội thoại.

# Custom commands
**Claude Code** đi kèm với các **built-in commands** (lệnh tích hợp sẵn) mà bạn có thể truy cập bằng cách gõ dấu `/`. Tuy nhiên, bạn cũng có thể tự định nghĩa **custom commands** (lệnh tùy chỉnh) để tự động hóa các tác vụ lặp đi lặp lại trong workflow phát triển phần mềm của mình.


## Create Custom Command

Để tạo một custom command, bạn cần thiết lập đúng **folder structure** trong project:

1. Tìm thư mục `.claude` trong **root directory** của project
2. Tạo một thư mục con tên là `commands` bên trong `.claude`
3. Tạo một file Markdown mới với tên bạn muốn đặt cho command (ví dụ: `audit.md`)

> ⚠️ Tên file chính là tên command.  
> Ví dụ: `audit.md` sẽ tạo ra command `/audit`.


## Command Audit

Giả sử bạn muốn tạo một command để **audit dependency** của project nhằm phát hiện và xử lý các lỗ hổng bảo mật (security vulnerabilities).

Command `/audit` có thể thực hiện 3 bước:

1. Chạy `npm audit` để phát hiện các package có lỗ hổng
2. Chạy `npm audit fix` để tự động update các dependency có thể vá được
3. Chạy test suite để đảm bảo việc update không gây **breaking changes**

Điều này giúp bạn đóng gói toàn bộ quy trình kiểm tra bảo mật thành một command duy nhất.

Sau khi tạo file command xong, bạn cần **restart Claude Code** để hệ thống load và register command mới.

![[Pasted image 20260303163748.png]]

## Custom Command Có Tham Số (Arguments)

Custom command có thể nhận tham số thông qua placeholder `$ARGUMENTS`. Điều này giúp command trở nên **dynamic**, linh hoạt và tái sử dụng được.

Ví dụ: file `write_tests.md` có thể chứa nội dung:

```
Write comprehensive tests for: $ARGUMENTS

Testing conventions:
* Use Vitests with React Testing Library
* Place test files in a __tests__ directory in the same folder as the source file
* Name test files as [filename].test.ts(x)
* Use @/ prefix for imports

Coverage:
* Test happy paths
* Test edge cases
* Test error states
```

Sau đó bạn có thể gọi command:

```
/write_tests the use-auth.ts file in the hooks directory
```

Trong trường hợp này:

- `$ARGUMENTS` sẽ nhận giá trị `"the use-auth.ts file in the hooks directory"`
- Claude sẽ sử dụng chuỗi này làm **context input** để thực thi tác vụ

Lưu ý:  
Argument không bắt buộc phải là file path. Bạn có thể truyền bất kỳ string nào để cung cấp thêm ngữ cảnh (context) hoặc yêu cầu cụ thể.

## Lợi Ích Chính
 1. Automation: Tự động hóa các workflow lặp lại thành một command duy nhất.
2. Consistency: Đảm bảo mọi developer trong team đều thực hiện cùng một quy trình (chuẩn hóa quy ước code, testing, deployment...).

3. Context Control: Cung cấp instruction rõ ràng cho Claude về coding convention, architecture, cấu trúc thư mục, tiêu chuẩn test...
4. Flexibility: Sử dụng arguments để command có thể áp dụng cho nhiều input khác nhau.
## Ứng Dụng Thực Tế
Custom commands đặc biệt hữu ích cho các workflow mang tính project-specific như:
- Chạy test suite
- Deploy code
- Generate boilerplate theo coding convention của team
- Refactor module theo pattern nhất định
- Kiểm tra security hoặc performance

Nói cách khác, custom command giúp bạn biến Claude Code thành một công cụ tự động hóa phù hợp với **SDLC (Software Development Lifecycle)** và tiêu chuẩn kỹ thuật riêng của dự án.

# MCP servers with Claude Code

Bạn có thể **mở rộng khả năng của Claude Code** bằng cách tích hợp các **MCP Server (Model Context Protocol)**.

**MCP Server** là các dịch vụ chạy **cục bộ trên máy (local)** hoặc **từ xa (remote server)**, giúp cung cấp thêm **tool, API và khả năng xử lý mới** cho Claude — những thứ mà mặc định Claude không có sẵn.

Khi kết nối với các **MCP Server**, Claude có thể:
- Truy cập và thao tác với **các công cụ bên ngoài (external tools)**
- Tương tác với **API hoặc dịch vụ backend**
- Thực hiện các **tác vụ tự động hóa (automation tasks)**
- Mở rộng **context và capability** của hệ thống AI

Nói cách khác, **MCP đóng vai trò như một giao thức chuẩn** giúp Claude **kết nối với các hệ thống khác** và sử dụng chúng như **plugin hoặc tool tích hợp** trong môi trường phát triển.

Một trong những **MCP server** phổ biến nhất hiện nay là **Playwright**, cho phép **Claude** có khả năng **điều khiển trực tiếp một trình duyệt web**.

Điều này mở ra nhiều khả năng mạnh mẽ cho các **workflow phát triển web (web development workflows)**. 

## Installing the Playwright MCP Server

Để thêm **máy chủ Playwright** vào **Claude Code**, hãy chạy lệnh này trong **terminal của bạn**.
`
```
claude mcp add playwright npx @playwright/mcp@latest
```

Lệnh này thực hiện hai việc:
1. Đặt tên cho máy chủ MCP là **"playwright"**.
2. Cung cấp lệnh để khởi động máy chủ đó cục bộ trên máy của bạn.

## Managing Permissions

Khi bạn lần đầu sử dụng các công cụ máy chủ **MCP**, **Claude** sẽ yêu cầu quyền cho mỗi lần sử dụng. Nếu bạn cảm thấy phiền vì các thông báo xin quyền này, bạn có thể **phê duyệt trước máy chủ** bằng cách chỉnh sửa cài đặt của mình. Hãy mở tệp `.claude/settings.local.json` và thêm máy chủ vào mảng `allow`:


```
{ "permissions": 
	{ 
		"allow": ["mcp__playwright"], 
		"deny": [] 
	} 
}

```

Lưu ý **hai dấu gạch dưới** trong `mcp__playwright`. Điều này cho phép **Claude** sử dụng các công cụ **Playwright** mà **không cần phải xin phép mỗi lần**.

## Practical Example: Improving Component Generation

Đây là một ví dụ thực tế về cách **Playwright MCP server** có thể cải thiện quy trình phát triển của bạn. Thay vì phải tự mình kiểm thử và tinh chỉnh prompt thủ công, bạn có thể để Claude:

- Mở trình duyệt và truy cập vào ứng dụng của bạn
- Tạo một component thử nghiệm
- Phân tích kiểu dáng hiển thị (visual styling) và chất lượng mã
- Cập nhật prompt tạo component dựa trên những gì nó quan sát được
- Kiểm tra prompt đã cải thiện bằng một component mới

Ví dụ, bạn có thể yêu cầu Claude:

> “Hãy truy cập vào `localhost:3000`, tạo một component cơ bản, xem lại phần styling, và cập nhật prompt tạo component tại `@src/lib/prompts/generation.tsx` để tạo ra các component tốt hơn trong tương lai.”

Claude sẽ sử dụng các công cụ trình duyệt để tương tác với ứng dụng của bạn, kiểm tra kết quả được tạo ra, sau đó chỉnh sửa file prompt nhằm khuyến khích tạo ra các thiết kế độc đáo và sáng tạo hơn.

## Results and Benefits

Trong thực tế, cách tiếp cận này có thể dẫn đến kết quả tốt hơn đáng kể. Thay vì các dải chuyển màu tím-sang-xanh chung chung và những mẫu **Tailwind** tiêu chuẩn, Claude có thể điều chỉnh prompt để khuyến khích:

- Các dải chuyển màu hoàng hôn ấm áp (cam → hồng → tím)
- Chủ đề độ sâu của đại dương (xanh teal → xanh lục ngọc → xanh cyan)
- Thiết kế bất đối xứng và các thành phần chồng lấp lên nhau
- Khoảng cách sáng tạo và bố cục không theo quy ước

Ưu điểm then chốt là Claude có thể nhìn thấy **kết quả hiển thị trực quan thực tế**, chứ không chỉ là mã nguồn. Nhờ vậy, nó có thể đưa ra những quyết định chính xác và có cơ sở hơn nhiều khi cải thiện phần kiểu dáng (styling).

## Exploring Other MCP Servers

Playwright chỉ là một ví dụ về những gì có thể làm được với các máy chủ MCP. Hệ sinh thái này bao gồm các máy chủ dành cho:

- Tương tác với cơ sở dữ liệu
- Kiểm thử và giám sát API
- Các thao tác với hệ thống tệp
- Tích hợp dịch vụ đám mây
- Tự động hóa các công cụ phát triển

Hãy cân nhắc khám phá các máy chủ MCP phù hợp với nhu cầu phát triển cụ thể của bạn. Chúng có thể biến Claude từ một trợ lý viết mã thành một đối tác phát triển toàn diện có thể tương tác với toàn bộ chuỗi công cụ (toolchain) của bạn.

# Github integration

**Claude Code** cung cấp một **tích hợp GitHub chính thức** cho phép Claude chạy bên trong **GitHub Actions**. Tích hợp này cung cấp hai quy trình làm việc chính: **hỗ trợ mention cho issues và pull requests**, và **tự động đánh giá (review) pull request**.

### Setting Up the Integration

Để bắt đầu, hãy chạy lệnh **/install-github-app** trong Claude. Lệnh này sẽ hướng dẫn bạn qua quá trình thiết lập:

- Cài đặt ứng dụng **Claude Code** trên GitHub
- Thêm **API key** của bạn
- Tự động tạo một **pull request** chứa các tệp workflow

![[Pasted image 20260307090919.png]]

Pull request được tạo ra sẽ thêm **hai GitHub Actions** vào repository của bạn. Sau khi được merge, bạn sẽ có các tệp workflow trong thư mục **.github/workflows**.


### Default GitHub Actions

Việc tích hợp này cung cấp hai quy trình làm việc chính:

**Mention Action**  
Bạn có thể nhắc đến Claude trong bất kỳ issue hoặc pull request nào bằng cách sử dụng **@claude**. Khi được nhắc đến, Claude sẽ:
- Phân tích yêu cầu và tạo kế hoạch thực hiện nhiệm vụ
- Thực hiện nhiệm vụ với quyền truy cập đầy đủ vào codebase của bạn
- Trả lời trực tiếp trong issue hoặc pull request với kết quả

**Pull Request Action**  
Mỗi khi bạn tạo một pull request, Claude sẽ tự động:
- Xem xét các thay đổi được đề xuất
- Phân tích tác động của các chỉnh sửa
- Đăng một báo cáo chi tiết trong pull request.

### Customizing the Workflows

Sau khi merge **pull request** ban đầu, bạn có thể tùy chỉnh các tệp **workflow** để phù hợp với nhu cầu của dự án. Dưới đây là cách cải thiện **mention workflow**:

#### Adding Project Setup

Trước khi Claude chạy, bạn có thể thêm các bước để chuẩn bị môi trường:

```
- name: Project Setup
  run: |
    npm run setup
    npm run dev:daemon
```

#### Custom Instructions

Cung cấp cho Claude ngữ cảnh về cách thiết lập dự án của bạn:

```
custom_instructions: |
  The project is already set up with all dependencies installed.
  The server is already running at localhost:3000.
  Logs from it are being written to logs.txt.
  If needed, you can query the db with the 'sqlite3' cli.
  If needed, use the mcp__playwright set of tools to launch a browser and interact with the app.
```

Dịch nghĩa:

```
Dự án đã được thiết lập sẵn với tất cả các dependency đã cài đặt.
Máy chủ đã chạy tại localhost:3000.
Log của hệ thống đang được ghi vào file logs.txt.
Nếu cần, bạn có thể truy vấn cơ sở dữ liệu bằng công cụ dòng lệnh 'sqlite3'.
Nếu cần, hãy sử dụng bộ công cụ mcp__playwright để khởi chạy trình duyệt và tương tác với ứng dụng.
```

#### MCP Server Configuration

Bạn có thể cấu hình các **MCP server** để cung cấp thêm khả năng cho Claude:

```
mcp_config: | 
{ "mcpServers": { "playwright": { "command": "npx", "args": [ "@playwright/mcp@latest", "--allowed-origins", "localhost:3000;cdn.tailwindcss.com;esm.sh" ] } } 
}
```

#### Tool Permissions

Khi chạy Claude trong GitHub Actions, bạn phải **liệt kê rõ ràng tất cả các công cụ được phép sử dụng**. Điều này đặc biệt quan trọng khi dùng các **MCP server**.

Ví dụ:

```
allowed_tools: "Bash(npm:*),Bash(sqlite3:*),mcp__playwright__browser_snapshot,mcp__playwright__browser_click,..."
```

Không giống như môi trường phát triển cục bộ (local development), **GitHub Actions không có cách tắt hoặc bỏ qua việc cấp quyền nhanh**. Mỗi công cụ từ mỗi MCP server **phải được liệt kê riêng lẻ**.

#### Best Practices

Khi thiết lập tích hợp Claude với GitHub:
- Bắt đầu với các **workflow mặc định**, sau đó **tùy chỉnh dần dần**.
- Sử dụng **custom instructions** để cung cấp **bối cảnh riêng của dự án**.
- **Chỉ định rõ quyền sử dụng công cụ** khi dùng MCP servers.
- **Kiểm tra workflow với các tác vụ đơn giản trước**, sau đó mới đến các tác vụ phức tạp.
- Cân nhắc **nhu cầu cụ thể của dự án** khi cấu hình thêm các bước bổ sung.


Tích hợp GitHub giúp **Claude chuyển từ một trợ lý phát triển** thành **một thành viên tự động của nhóm**, có thể xử lý nhiệm vụ, đánh giá code và cung cấp insight **trực tiếp trong workflow GitHub của bạn**.
